## F1: Token 统计 by daily / monthly

**用户原始描述**:
> 加 token 统计，by daily/monthly

**意图解读**:
- 按日聚合（已有） + 按月聚合（缺）
- 范围：跨所有 provider？or 单 provider？or 顶层 header 摘要 + 详情展开？
- 展示形态：数字 / 列表 / 图表？文本为主？需要一块曲面？

**已有数据基础**:
- `UsageHistory` model 已有 daily totals per provider (per `docs/superpowers/specs/2026-06-30-token-king-fork设计.md` 引用)
- 但 monthly aggregation 没看到实现
- 持久化: `UsageHistory.fetchAndPersistIfNeeded()` 类机制已有

**需要决策**:
1. monthly aggregation 怎么算 — 跟现有 daily 累计还是按月独立采样？
2. 历史数据窗口 — 永久保存 vs 滚动 12 个月？
3. UI 入口 — 顶层 header 加 "本月 ≈ XX token" 摘要 or 新建子菜单？or 状态栏 widget？
4. CSV/JSON 导出 — 要不要？开发调试 / 用户本地备份都常见诉求

**潜在拆解**:
- F1a: 单 provider 详情 → daily/monthly token 列表 (纯文本)
- F1b: 顶层 header → 当月总 token 摘要
- F1c: 数据导出 (csv/json)

## F2: 金额换算比率（API 价 vs 订阅价）

**用户原始描述**:
> 加金额转比，比如 199 元的 kimi 订阅，按照 api 价格，我当前花了多钱

**意图解读**:
- 用户视角："订阅 vs 按量付费" 的经济学比较
- 例：¥199/月 kimi 订阅；如果按 API 单价算，要 ¥500。用户就亏了；按 ¥150，订阅就赚了
- 应该是 top-level 头部信息的一部分（跟 R1 / F1 一起显示）

**已有数据基础**:
- `CurrencyFormatter` 当前 USD ↔ RMB 汇率换算已有（`currentRate`）
- 单价数据？每个 provider 的 per-token API 单价 — **完全没有**，需要找来源
- subscription 价格 — `ProviderSubscriptionPresets` 已有（按 cnyCost 显示）

**需要决策**:
1. API 单价来源 — 硬编码（不准）+ 定期拉（复杂）+ 用户手动设置（灵活）
2. 计算口径 — 这次 fetch 的 token 数 × 单价 vs 历史累计 × 单价 vs 月度聚合 × 单价？
3. 展示形态 — "API 价 ¥450 / 订阅 ¥199, 订阅省 ¥251" 还是 "(API 价 / 订阅价) × 100 = 226%, 超出 1.26x"？
4. 跨 provider 还是单 provider？还是每个 provider 一个独立判断？

**潜在拆解**:
- F2a: 提供一个"硬编码 API 单价表"基础设施（先有数据，再说换算）
- F2b: 单 provider 头部加一行 API 价 vs 订阅价
- F2c: 跨 provider 汇总

## F3: 使用记录（今日 / 本周进度）

**用户原始描述**:
> 加使用记录，还是 199 元的 kimi 订阅，我今天花了 2 个 5小时限额，分别是 1%, 15%，上周总花费周额度的 54%，这周是 35%

**意图解读**:
- "5小时限额" = 5小时 quota 桶（per-hour / 5-hour rolling window）
- 用户想要的展示：
  - 今天已经触达的 quota 桶（已经触达 1% + 15% — 这两个加起来 16%）
  - 本周累计百分比（上周 54%, 这周 35%）
- 信息已经在 `UsageHistory.dailyTotals` 里，但**当前 UI 没把这些展示给用户**

**已有数据基础**:
- `DetailedUsage` 各 provider 已经返回 hour-bucket utilization（Kimi 应该已经有五分钟限额吧？or 五小时？看 provider 实现）
- `StatusBarController` 已经能展示 "5h: 0%" "Weekly: 74%" 这种摘要（之前 P0 audit 修过）
- 但用户具体描述的"今天几个桶分别多少%/本周进度"这种**明细**没有 sub-menu 视角

**需要决策**:
1. 何时清 5h 桶的"今天已触达" — 5 小时滑窗 vs 凌晨 0 点硬清？→ 影响展示准确性
2. 展示粒度 — 单 provider 详情里看？or 跨 provider 比较？
3. "本周"定义 — 自然周（周一 0 点重置）vs 滚动 7 天？跟 provider 的"weekly reset" 同步 vs 跟公历周同步？

**潜在拆解**:
- F3a: 单 provider 详情 → 当日桶明细 + 本周累计
- F3b: 跨 provider → 今日 / 本周 总用度 (走 F4)

## F4: 全局统计模块（顶层入口）

**用户原始描述**:
> 我想要个使用全局统计

**意图解读**:
- 单 provider 详情是 Atomic 的，但用户想看"全景"
- 例：今天总花了多少、本周总进度、跨 provider 月度环比
- 应该是 header 区域 + 顶菜单 / 子菜单 集成

**与上面三个需求的区别**:
- F1: 数据基础设施（by daily/monthly）
- F2: 价格换算
- F3: 单 provider 使用记录
- F4 = F1 + F2 + F3 + 跨 provider 聚合视图

**潜在形态**:
- 一个新的子菜单 "📊 全局统计"，或
- 状态栏菜单 header 区域加新一行聚合信息

**与现有额度状态的关系**:
- 现在的 header "额度状态: ¥1329/月" 是 subscription 总价统计
- F4 是**用量**统计
- 应该并排展示，或者把 header 区扩展为两段

## 共同的依赖 / 风险

| 风险 | 影响 F1-F4 多少 | 需要先解吗 |
|---|---|---|
| 单 provider 详情入口/位置不统一 | F3 受影响最大 | 先 audit `ProviderMenuBuilder.createProviderSubmenu*` 各 provider 的渲染结构 |
| 跨 provider 汇总需要 `UsageHistory` 扩展（加 provider × daily） | F1 + F4 受影响 | 是，需要先扩 data model |
| API 单价表无 source-of-truth | F2 必须先解 | **是**，hardcode 起步先 |
| 顶部 header 当前已经塞满了 "额度状态 / 重复警告 / Orphaned" | F1+F4 可能没空间 | 先看 header 区布局 |
